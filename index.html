BI Maturity Level Concept by Ventera ¹

A well-designed data architecture should reach at least maturity level 4 to deliver expected benefits, increase reporting productivity, and avoid expensive maintenance.

Level 5
Data Driven Predictions
Advanced predictive analysis • Big data capabilities • Identifies patterns
Level 4
Self Service Analysis
User driven analysis • Explored across dimensions • In depth correlations
Level 3
Management Dashboards
Executive level reports • Drillable reports • Insights using trends
Level 2
Operational Reports
Data from transactional systems • Reports on "What's happening"
Level 1
Unorganized
As needed data reports • Heavy use of localized spreadsheets • Single point of failure

Portfolio & Core Concepts

A glimpse into my reporting and BI architecture mindset

UX/UI Design

I try to deliver more than just developing dashboards. To me, developing Power BI products with maturity-level 4 means creating a true single source of truth in one intuitive controlling software based on one semantic model (one single .pbix-File).

This makes maintenance exponentially easier and usability for all users across the company a calming rejoice.

KPI Developing

As a controller this always has been the core of my profession.

Metrics are determining the entire process of building an analytical data architecture, so it is crucial to create a KPI-system first.

This BI process often got underestimated by analysts without strong background in finance and controlling. It is necessary to precisely define stakeholder requirements by maintaining data consistency at the same time.

Process Mining

Process Mining bridges the gap between data analytics and business operations. I prefer to use Figma to visually map and mine these workflows, acting as the crucial preliminary stage before developing any KPIs.

This allows us to identify bottlenecks, inefficiencies, and deviations from the ideal path, enabling data-driven process optimization and a deeper understanding of operational reality.

Data Warehousing

Lean and functional data modeling asks for tables of a high information density.
For this there is no better approach than the medallion architecture from Apache Spark, which is now adopted by the notebook functionality of Microsoft Fabric with T-SQL, making it the perfect tool for analytical warehousing.


Picture: customer cohort analysis with [cross-sectional] follow up analysis

Data Modeling

Being able to have a single source of truth in one semantic model means creating a smart and efficient data model which allows a smooth navigation and filtering ability for the user and stakeholder.
This can only be obtained by a smart data warehousing, which can not be viewed apart from data modeling on BI maturity level 4.


Picture: for the user it is possible to filter the entire dashboard model by all dimensions, like customer or item, in the very same UX/UI they use for general analytics.

Finance Controlling

Operational controlling and finance are often disconnected. To ensure that operational controlling remains compatible with official reports like balance sheets, P&L, and cash flow, accounting software should always be integrated and reconciled.

Breaking out of the usual Excel landscape also allows for many quick wins within the finance department, as demonstrated here in a project for Tipico in Malta.
